# everything-claude-code 레포지토리 분석

이 레포지토리는 **Anthropic 해커톤 우승자**가 10개월 이상 Claude Code를 사용하면서 축적한 **실전 검증된 설정 모음집**입니다.

---

## 핵심 개념: 6가지 구성 요소

```
everything-claude-code/
├── agents/       # 특화된 서브에이전트 (위임용)
├── skills/       # 워크플로우 정의 및 도메인 지식
├── commands/     # 슬래시 커맨드 (빠른 실행)
├── rules/        # 항상 따라야 하는 가이드라인
├── hooks/        # 트리거 기반 자동화
├── mcp-configs/  # MCP 서버 설정
└── plugins/      # 플러그인 생태계 문서
```

---

## 1. Agents (에이전트) - 위임 전문가

**역할**: 특정 작업에 특화된 서브에이전트로, 메인 AI가 작업을 위임할 때 사용

| 에이전트 | 역할 |
|----------|------|
| `planner.md` | 기능 구현 계획 수립 |
| `architect.md` | 시스템 설계 결정 |
| `tdd-guide.md` | 테스트 주도 개발 가이드 |
| `code-reviewer.md` | 코드 품질 및 보안 검토 |
| `security-reviewer.md` | 취약점 분석 |
| `build-error-resolver.md` | 빌드 에러 해결 |
| `e2e-runner.md` | Playwright E2E 테스트 |
| `refactor-cleaner.md` | 데드 코드 정리 |
| `doc-updater.md` | 문서 동기화 |

**에이전트 정의 예시**:

```markdown
---
name: code-reviewer
description: Reviews code for quality, security, and maintainability
tools: Read, Grep, Glob, Bash
model: opus
---

You are a senior code reviewer...
```

## 2. Skills (스킬) - 도메인 지식 & 워크플로우

**역할**: 특정 도메인의 지식이나 워크플로우를 정의하여 에이전트나 커맨드가 참조

| 스킬 | 내용 |
|------|------|
| `coding-standards.md` | 언어별 베스트 프랙티스 |
| `backend-patterns.md` | API, DB, 캐싱 패턴 |
| `frontend-patterns.md` | React, Next.js 패턴 |
| `tdd-workflow/` | TDD 방법론 |
| `security-review/` | 보안 체크리스트 |
| `clickhouse-io.md` | ClickHouse 분석 |

**TDD 워크플로우 예시**:

```markdown
# TDD Workflow
1. Define interfaces first (인터페이스 정의)
2. Write failing tests (RED) (실패 테스트 작성)
3. Implement minimal code (GREEN) (최소 코드 구현)
4. Refactor (IMPROVE) (리팩토링)
5. Verify 80%+ coverage (커버리지 검증)
```

## 3. Commands (커맨드) - 슬래시 명령어

**역할**: `/커맨드명`으로 빠르게 실행할 수 있는 명령어

| 커맨드 | 기능 |
|--------|------|
| `/tdd` | 테스트 주도 개발 시작 |
| `/plan` | 구현 계획 수립 |
| `/e2e` | E2E 테스트 생성 |
| `/code-review` | 코드 품질 검토 |
| `/build-fix` | 빌드 에러 수정 |
| `/refactor-clean` | 데드 코드 제거 |
| `/test-coverage` | 커버리지 분석 |
| `/update-docs` | 문서 동기화 |

## 4. Rules (규칙) - 항상 따라야 하는 가이드라인

**역할**: 모든 작업에서 항상 적용되는 규칙 (자동 적용)

| 규칙 | 내용 |
|------|------|
| `security.md` | 필수 보안 검사 |
| `coding-style.md` | 불변성, 파일 구조 |
| `testing.md` | TDD, 80% 커버리지 요구 |
| `git-workflow.md` | 커밋 형식, PR 프로세스 |
| `agents.md` | 서브에이전트 위임 시점 |
| `performance.md` | 모델 선택, 컨텍스트 관리 |

**규칙 구조 권장사항**:

```
~/.claude/rules/
├── security.md      # 하드코딩된 시크릿 금지
├── coding-style.md  # 불변성, 파일 제한
└── testing.md       # TDD, 커버리지 요구사항
```

---

## 5. Hooks (훅) - 이벤트 기반 자동화

**역할**: 도구 실행 전/후에 자동으로 트리거되는 자동화

**트리거 타입**:
- `PreToolUse`: 도구 실행 전
- `PostToolUse`: 도구 실행 후
- `Stop`: 작업 종료 시

**console.log 경고 훅 예시**:

```json
{
  "matcher": "tool == \"Edit\" && tool_input.file_path matches \"\\.(ts|tsx|js|jsx)$\"",
  "hooks": [{
    "type": "command",
    "command": "#!/bin/bash\ngrep -n 'console\\.log' \"$file_path\" && echo '[Hook] Remove console.log' >&2"
  }]
}
```

## 6. MCP Configs (MCP 설정)

**역할**: 외부 서비스와 연동하는 MCP 서버 설정

**포함된 서버**: GitHub, Supabase, Vercel, Railway 등

**중요 경고 - 컨텍스트 윈도우 관리**:

| 상태 | 권장 |
|------|------|
| MCP 전체 활성화 | 200k → 70k로 축소될 수 있음 |
| 설정된 MCP | 20-30개 |
| 프로젝트별 활성화 | 10개 미만 |
| 활성 도구 | 80개 미만 |

## 핵심 인사이트

### 1. 모듈화 원칙
각 구성 요소가 **단일 책임**을 가지며, 필요한 것만 선택적으로 사용 가능

### 2. 위임 패턴
복잡한 작업은 **전문 에이전트에게 위임**하여 품질 향상

### 3. 컨텍스트 관리가 핵심
MCP를 무분별하게 활성화하면 **컨텍스트 윈도우가 급격히 축소**됨

### 4. 실전 검증
Anthropic 해커톤 우승작 `zenith.chat` 개발에 실제 사용된 설정들

## 활용 방법

```bash
# 1. 클론
git clone https://github.com/affaan-m/everything-claude-code.git

# 2. 필요한 것만 복사
cp everything-claude-code/agents/*.md ~/.claude/agents/
cp everything-claude-code/rules/*.md ~/.claude/rules/
cp everything-claude-code/commands/*.md ~/.claude/commands/

# 3. hooks.json → settings.json에 추가
# 4. MCP 설정에서 API 키 교체

| 구분 | 현재 User Rules (한국형) | Everything-Claude-Code |

## 2. 구성 요소별 비교

### 2.1 에이전트 시스템

| 항목 | 현재 시스템 | Everything-Claude-Code |
|------|------------|------------------------|
| **구조** | 상급 팀(총괄/PM/관리자/감사) + 프로젝트 팀 | 기능별 서브에이전트 |
| **역할** | 백그라운드 모니터링, 보고서 생성 | 작업 위임, 코드 품질 검증 |
| **에이전트 예시** | 총괄, PM, 관리자, 감사, 코딩팀 | planner, architect, code-reviewer, security-reviewer |
| **슈퍼바이저** | 최고+시니어 슈퍼바이저 구조 | 없음 (단일 에이전트) |
| **특징** | MCP 활용 규칙 내장, 논쟁 기반 협업 | 도구 제한, 모델 지정 가능 |

**차이점 분석**:
- 현재 시스템: **거시적 관리** 중심 (프로젝트 상태 추적, 보고서)
- ECC: **미시적 실행** 중심 (코드 리뷰, 테스트, 리팩토링)

## 3. 강점/약점 비교

### 현재 시스템 강점
1. **문서/파일 관리 체계**: 번호 + 시각 + 메타태그로 추적 가능
2. **자동 로그 시스템**: 백그라운드에서 변경사항 자동 기록
3. **에이전트 계층 구조**: 상급 팀 → 프로젝트 팀 명확한 위계
4. **MCP 활용 규칙**: 검증 데이터 기반 작업 강제
5. **환각 방지**: "뇌피셜 금지" 명시적 규칙

### 현재 시스템 약점
1. **개발 워크플로우 부재**: TDD, 코드 리뷰 프로세스 없음
2. **품질 검증 부재**: 커버리지, 보안 검토 규칙 없음
3. **실행 커맨드 미구축**: `_COMMAND_LIBRARY/` 폴더 없음
4. **Hooks 미활용**: 도구 실행 전/후 자동화 없음

### Everything-Claude-Code 강점
1. **개발 워크플로우**: TDD, 코드 리뷰, E2E 테스트
2. **품질 검증**: 80% 커버리지, 보안 검토
3. **Hooks 자동화**: 도구 실행 시 자동 검증
4. **에이전트 위임**: 복잡한 작업을 전문 에이전트에게 위임

### Everything-Claude-Code 약점
1. **Cursor 미지원**: Claude Code (CLI) 전용
2. **문서 관리 부재**: 파일명 규칙, 버전 관리 없음
3. **프로젝트 관리 부재**: 상태 추적, 보고서 시스템 없음
4. **한국어 미지원**: 영어 전용

## 📋 개발 품질 규칙 (ECC 통합)

### TDD 워크플로우
1. 인터페이스 먼저 정의
2. 실패 테스트 작성 (RED)
3. 최소 코드 구현 (GREEN)
4. 리팩토링 (IMPROVE)
5. 80% 이상 커버리지 검증

### 코드 리뷰 체크리스트
- [ ] 보안 취약점 검토

- [ ] 에러 핸들링 적절함

### Git 커밋 규칙
- 형식: `[타입] 파일명: 변경사항`
- 타입: feat, fix, refactor, test, docs, chore

**역할**: 보안 취약점 분석
**트리거**: PR 전, 배포 전

### TDD Guide
**역할**: 테스트 주도 개발 가이드
**트리거**: 새 기능 개발 시
```

## 7. 최종 권장 사항

| 판단 | 결론 |
|------|------|
| **완전 교체** | ❌ 비추천 (현재 시스템의 강점 손실) |
| **부분 통합** | ✅ 강력 추천 |
| **무시** | ❌ 비추천 (개발 품질 향상 기회 손실) |

### 우선 적용 순서

1. **즉시**: `_COMMAND_LIBRARY/` 폴더 구축
2. **1단계**: TDD 워크플로우 + 코드 리뷰 규칙 추가
3. **2단계**: 보안 검토 규칙 + Git 워크플로우 추가
4. **3단계**: 프로젝트 팀에 Code Reviewer, Security Reviewer 에이전트 추가

현재 시스템의 **문서/프로젝트 관리 강점**을 유지하면서, ECC의 **개발 품질 검증 강점**을 통합하는 하이브리드 접근이 가장 효과적입니다.
