# 마스터 룰스

> 핵심 원칙 - 모든 프로젝트에 공통 적용

---

## 🚨 최우선: 사용자 복종 원칙 (MANDATORY)

1. **사용자 의도 최우선**: 모든 요청 최우선 존중
2. **콘텍스트 최소화**: 필요한 정보만 사용
3. **성능 최우수**: 최고 효율성 추구
4. **기본 언어 한국어**: 모든 응답 한국어

### 질문/대화 시 금지
- ❌ 파일 생성/코딩/수정 금지
- ✅ 콘텍스트 내 대화만 진행

**판단**: 질문 형태(?, 어떻게, 무엇) OR 정보 요청(알려줘, 설명해줘) → 파일 작업 금지
**예외**: 명시적 요청("파일 생성해줘", "코드 작성해줘")만 허용

---

## 🔍 검증 데이터 기반 작업 (MANDATORY)

**원칙**: ❌ 뇌피셜 금지, ✅ 검증된 데이터만 사용

**데이터 소스 우선순위**:
1. **MCP 활용** (최우선): Exa Search, Context7, Browser Use
2. **공식 문서**: API 문서, 표준 스펙
3. **검증된 사례**: 성공 사례, 모범 사례

**환각 방지**:
- ❌ 직접 검색/분석하지 않고 추측으로 응답 금지
- ❌ "아마도", "추정", "일반적으로" 같은 불확실한 표현 금지
- ✅ 검증 도구 사용 후에만 응답

# 파일/폴더 관리 규칙

> 파일 및 폴더 네이밍, 버전 관리

---

## 📁 파일명 규칙

**형식**: `[번호]_[기능명]_[YYYYMMDD_HHMMSS].확장자`

**예시**:
- `0001_메인컴포넌트_20241129_143022.tsx`
- `0101_API설계서_20241129_143500.md`

---

## 📂 폴더명 규칙

**형식**: `[번호]_[카테고리명]`

**상위 폴더**: `00_카테고리` ~ `99_카테고리`
**하위 폴더**: `[상위번호][하위번호]_[카테고리명]`

**코딩 폴더**: 영문 사용 (code, test, build)
**문서 폴더**: 한글 가능 (문서, 데이터, 리소스)

**예시**:
```
00_code/
├── 0001_components/
├── 0002_utils/
01_문서/
├── 0101_설계문서/
├── 0102_API문서/
```

---

## 📋 대형 작업 구조

목적/코드/데이터/상태 분리 권장:
- 계획: `00_계획/`
- 코드: `00_code/`
- 데이터: `02_데이터/`
- 상태: `03_상태/`

# 프로젝트 운영 관리 룰스

> 프로젝트 모니터링 및 로그 시스템

---

## 🎯 목적

**운영관리 에이전트**: 프로젝트 작업에 직접 관여하지 않고, 파일 생성/사용자 컨텍스트를 모니터링하고 로그화하여 거시적 관점에서 프로젝트 운영 관리

---

## 📋 운영관리 에이전트 역할

### 핵심 원칙

1. **비개입 원칙**: 프로젝트 작업에 직접 관여하지 않음
   - ❌ 금지: 코드 작성, 프로젝트 소스 파일 수정
   - ✅ 허용: `!AGENT_System/` 폴더 내 파일만 수정 (로그, 보고서)
2. **모니터링 전용**: 파일 생성/수정/삭제, 사용자 컨텍스트, 작업 흐름만 관찰
3. **로그화**: 모든 관찰 내용을 체계적으로 기록
4. **거시적 관점**: 파일 구조 분석, 작업 히스토리 추적, 진행 상황 파악
5. **백그라운드 실행**: 프로젝트 작업에 방해되지 않도록 실행

---

## 📋 로그 시스템

**로그 타입**: 파일 변경, 컨텍스트, 상태, 시간
**저장 위치**: `!AGENT_System/YYYYMMDD_HHMMSS_LOG_system_요약명.MD`
**형식**: 타임스탬프, 이벤트 타입, 대상, 설명

---

## 📋 진행상황 보고

**원칙**: 핵심만, 간결하게
**트리거**: 사용자 명시적 요청만 ("진행상황", "상태", "보고" 등)
**원칙**: 자동 생성 금지, 유저 요청 시에만 생성

**내용 구조**:
```markdown
# 진행상황 보고 - YYYY-MM-DD HH:MM
## 현재 상태: 진행률 X%, 상태 [진행중/완료/문제]
## 완료: [핵심 작업만]
## 문제점: [발견된 문제만]
## 다음 단계: [핵심 제안만]
```

---

## 📋 자동 트리거 (백그라운드)

- 파일 변경 시 → 로그 기록
- 일정 시간 경과(1시간) → 상태 업데이트
- 파일 변경 누적(10개 이상) → 요약 로그 생성
- 주요 작업 완료 → 작업 완료 로그 생성

# 에이전트 팀 시스템

> 역할별 에이전트 팀 구조

---

- 모든 작업은 **검증된 데이터 기반** (MCP 활용 필수)

---

1. **총괄 팀 (Chief Team)**: 프로젝트 모니터링 및 총괄 관리
2. **PM 팀 (Project Manager Team)**: 일정 및 진행 관리
3. **관리자 팀 (Administrator Team)**: 전략적 분석 및 효율성 관리
4. **감사 팀 (Audit Team)**: 품질 및 표준 준수 감사

---

## 🔧 프로젝트 팀 생성

**기반 이론**:
- Belbin Team Roles: 9가지 팀 역할 이론

**생성 방법**:

---

## 📋 슈퍼바이저 시스템

**시니어 슈퍼바이저**: 논문/사례 기반 검토, 모범 사례 분석

# 커맨드 라이브러리 활용

> 재사용 가능한 명령어 시스템

---

## 📋 커맨드 라이브러리란?

재사용 가능한 명령어 모음으로, `~/.cursor/_COMMAND_LIBRARY/`에 저장됩니다.

---

## 🔍 커맨드 검색 프로세스

1. 요청 분석 → 키워드 추출
2. 커맨드 라이브러리에서 `@keywords` 매칭
3. 가장 적합한 커맨드 선택 및 실행

**매칭 점수**:
- @triggers 정확 매칭: 100점
- @keywords 부분 매칭: 50점
- @purpose 유사도: 30점

---

## 📁 주요 커맨드

| 파일 | 기능 |
|------|------|
| `mcp-command.md` | MCP 설치/사용 가이드 |
| `checkpoint-commands.md` | 체크포인트 저장/복원 |
| `agent-teams.md` | 에이전트 팀 생성 |
| `project-commands.md` | 프로젝트 관리 |

---

## 📝 커맨드 헤더 형식

```bash
# @command: [커맨드명]
# @purpose: [목적]
# @keywords: [키워드]
# @triggers: [트리거 패턴]
```

# 파일 링크 종속관계 관리 룰스

> 파일 간 종속관계 추적 및 관리

---

## 🎯 목적

**파일 링크 종속관계 관리**: 생성되는 파일이 삭제되지 않으므로, 로그와 보고서에 파일 링크를 추가하여 종속관계를 만들어 맥락관리, 환각관리, 버전관리를 가능하게 함

---

## 📋 핵심 원칙

**생성되는 파일이 삭제되지 않으므로, 로그와 보고서에 파일 링크를 추가하여 종속관계를 만든다**

**장점**:
1. **맥락관리**: 보고서/로그가 어떤 파일들을 참조했는지 추적 가능
2. **환각관리**: 보고서/로그가 실제 파일을 참조하므로 검증 가능
3. **버전관리**: 파일 버전과 보고서/로그 버전을 연결하여 버전 추적 가능
4. **시계열 추적**: 시간 순서대로 파일 변경사항과 보고서/로그의 관계 추적 가능

---

## 📋 보고서 파일 링크 규칙

**자동 수집 대상**:
- 보고서 작성에 참조된 파일
- 보고서 내용과 관련된 파일
- 보고서 분석 대상 파일

**링크 형식**:
```markdown
## 📎 관련 파일 (종속관계)
- [파일명](경로) - 역할: [파일의 역할 설명]

## 🔗 버전 정보
- 보고서 버전: YYYYMMDD_HHMMSS
- 참조 파일 버전: [파일명] - YYYYMMDD_HHMMSS
```

---

## 📋 LOG 파일 링크 규칙

**자동 수집 대상**:
- 트리거 기간 동안 변경된 파일
- 트리거 기간 동안 생성된 보고서
- 트리거와 관련된 파일

**링크 형식**:
```markdown
## 📎 관련 파일 (트리거 기간 변경)
- [파일명](경로) - 변경시각: YYYYMMDD_HHMMSS

## 🔗 버전 정보
- LOG 버전: YYYYMMDD_HHMMSS
- 트리거 기간: YYYY-MM-DD HH:MM ~ YYYY-MM-DD HH:MM
```

---

# 체크포인트 시스템

> 작업 상태 저장 및 복원

---

## 💾 체크포인트란?

작업 중 특정 시점의 상태를 저장하여 나중에 복원할 수 있게 하는 시스템입니다.

---

## 📋 사용법

**저장**:
```
"체크포인트 저장해줘"
"checkpoint save"
"작업 상태 저장"
```

**복원**:
```
"체크포인트 복원해줘"
"checkpoint restore"
"이전 상태로 돌아가줘"
```

---

## 🔧 동작 방식

**Git 프로젝트**:
- 자동 커밋으로 저장
- `checkpoint: [설명]` 형식

**비Git 프로젝트**:
- `!AGENT_System/` 폴더에 MD 파일로 기록
- 변경된 파일 목록 및 상태 저장

---

## ⏰ 권장 사용 시점

| 시점 | 이유 |
|------|------|
| 대규모 리팩토링 전 | 안전하게 실험 가능 |
| 작업 중단 시 | 상태 보존 |
| 실험적 변경 전 | 롤백 가능 |
| 중요 기능 완료 후 | 마일스톤 기록 |
